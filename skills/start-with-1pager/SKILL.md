# Start with 1-Pager

## 这个 skill 是什么

用户的个人方法论 skill。把两套互补的工具串联使用：

- **金字塔思考**（SCQA + MECE）→ 定方向
- **MVP 执行** → 定颗粒度
- **完美主义检测** → 防拖延

**底层认知**：用户在 2026-05 总结过——自己的"表达问题"（自己懂但讲不清）本质是"思考问题"（在结构化之前就开始表达 / 做事）。这个 skill 解决根因。

核心一句：
> **完美主义想"怎么做"，金字塔思考想"做什么"。**
> **方向上认真，执行上糙一点。**

## 何时启用 / 何时不要启用

用户显式调用本 skill 名称时启动。常见说法：
- "用 start-with-1pager"
- "用我的方法论 / 用 1-pager 帮我想一下"

**不要**用于以下场景（这些场景结构化会显得冷血）：

| 场景 | 应该 |
|---|---|
| 用户在倾诉、聊天、情绪整理 | 先共情，不要结构化 |
| 用户说"听我说说 / 我难受 / 帮我捋一捋情绪" | 自然回应 |
| 私人记录、日记、随手笔记 | 让用户自由流淌 |

**判断口诀**：用户问"帮我想清楚 / 帮我开始"才用，说"听我说说"不用。

若当前工作区有 `AGENTS.md` 等行为规则（如个人记录仓库），遵循其优先级。

## 启动后：决定跑哪个 workflow

| 用户当前状态 | Workflow |
|---|---|
| 要开始新任务（项目 / 调研 / 博客 / 汇报 / 决策） | **A: Start with 1-pager** |
| 已有产出物，想审视表达结构 | **B: Review expression** |
| 卡在"我先想想 / 等准备好 / 我还没想清楚" | A + 完美主义提示 |
| 询问"我应该怎么开始" | A |

模糊时默认 A。真的判断不出来再问一次。

## Workflow A: Start with a 1-pager

### Step 0: 方向 vs 执行 检查

先问用户：**"你现在'想'的是方向还是执行？"**

- 想"做什么 / 为什么做 / 怎么衡量成功" → ✅ direction → 进 Step 1
- 想"用什么技术 / 用什么方法 / 用什么 prompt / 怎么调参数" → ❌ execution
  → 这是完美主义信号。提醒用户：**执行细节动手做就知道了，现在该想方向**。

### Step 1: 限时 1 小时写 1-pager

把这个模板给用户填（agent 不要替用户填实质内容，但可以提示该填什么）：

```markdown
# [任务名] 1-Pager

## 1. Background (S)
这件事在什么领域？我的受众/用户在乎什么？

## 2. Problem (C)
现状哪里不够好？为什么现在做？

## 3. Hypothesis (Q + A, 一句话)
我猜测的核心答案：___________

## 4. Approach (MECE 子任务)
- 子任务 1: ...
- 子任务 2: ...
- 子任务 3: ...
（2-4 个，相互独立、完全穷尽）

## 5. Success criteria
做到什么程度算成功？

## 6. Risks
可能哪里翻车？我最可能错的假设是什么？
```

用户写得太长/太详细时，及时打断。

### Step 2: 粗糙度检查（5 条判据）

1-pager 写完后跑这 5 条。**任何 ❌ 都是完美主义入侵**：

1. ⏱️ 用时 < 1 小时？
2. 📝 Hypothesis 只有 1 句话（不是几段）？
3. 🧩 Approach 是 MECE 的（不重不漏）？
4. 🤝 用户接受这版假设可能错、将来会更新？
5. 📏 总长 ≤ 1-2 页（不是 5 页详细方案）？

≥ 1 个 ❌：提醒 **"v0 就是粗糙的，目的是有起点，不是完美方案"**。

### Step 3: 交接到 MVP 执行

1. 选 Approach 里第一个子任务
2. 拆成 1-3 天能跑通的 v0
3. 明确写下"这个 MVP 要验证的逻辑是什么"
4. 提醒用户：执行中纠结某个技术细节时 → 拍一个粗糙版本，标记"待定"，继续做

## Workflow B: Review expression structure

用户分享产出物（文章草稿 / PPT / 演讲稿 / 重要消息 / 汇报开场）请你审视时，跑这 5 个检查：

### Check 1: Conclusion-first?

第一句 / 第一段 / 第一页是不是直接讲了结论 / 主张 / 答案？
- 藏在中间或最后 → 提醒重组

### Check 2: SCQA opening?（重要场合）

答辩 / 汇报 / 面试 / 跨领域分享，开场是否有：

- **S**: **受众的**语境（不是作者的语境）
- **C**: 受众也认可的痛点
- **Q**: 这次要回答的问题
- **A**: 一句话答案

⚠️ 当受众跨领域时（比如内容是 CS / 大模型，受众是经济学 / 管理学），**开场必须用受众的语言锚定**，不能上来就讲自己的方法。这是用户在 2026-05 答辩被批评的根因。

### Check 3: MECE 分组?

主要分点是不是"相互独立、完全穷尽"？

- ✅ "做了什么 / 效果如何 / 下一步"（按动作顺序，MECE）
- ❌ "做了什么 / 用户喜欢 / 技术难点"（三个维度混杂）

混杂 → 帮用户重新归类。

### Check 4: Pyramid depth?

是不是 "结论 → 2-5 个分论点 → 证据" 的三层结构？
- 每个分论点独立支持结论？
- 每个分论点有具体证据？

### Check 5: Compression test

让用户答：**"如果对方只能记住一句话，他会记什么？"**

- 答 "X 可以分成三类" → ❌ 没 insight，要重组
- 答 "X 的核心是 Y，因为 Z" → ✅ 有 insight

## 完美主义信号词典

用户说这些话时，识别并温和地引导：

| 信号词 | 真实含义 | 建议回应 |
|---|---|---|
| "我先想想" | 想不出来又不想动 | 限时 1 小时写 1-pager |
| "等我准备好" | 害怕暴露不完美 | "v0 就是粗糙的" |
| "我得先调研一下" | 用调研拖延决策 | 调研也要 timebox（2 小时够了） |
| "我还没想清楚" | 想错了维度（在想执行不是方向） | 跑 Step 0 检查 |
| "万一错了怎么办" | 把错当作否定 | "错是信息，不是否定" |
| "我想做到完美" | 终态思维 | "Done > perfect。先 v0，再 v1，再 v2" |
| 反复改一份文档 / 反复推翻方案 | 经典完美主义 | "现在是 v0，不是 v3" |

## 给 Agent 的执行提醒

1. **不要替用户思考实质内容**。1-pager 里的具体答案必须是用户写的——agent 只引导填什么字段、提醒粗糙度、检查 MECE。
2. **保持限时压力**。如果用户开始陷入细节，提醒"我们在 1-pager 阶段，不在执行阶段"。
3. **欢迎错误**。如果用户写的 hypothesis 听起来可能错，不要急着纠正——这就是 1-pager 的意义（有个粗糙起点）。下一次迭代再更新。
4. **不要在情绪场景启用**。如果用户在情绪低谷或纯倾诉中喊出本 skill 名，先确认："你现在是想用方法论开始干一件事，还是先聊聊感受？"
5. **结束时一句收尾**。任务跑完后，提醒用户："这个 1-pager 是 v0，做的过程中可以更新它。"
